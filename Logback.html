Logback

1.概念

1.logback是一种最流行的日志技术，用于在项目中完成对应的操作记录，与spring联系紧密

2.优势

3.关键组件

一.Logger

Logger 是应用代码里调用的对象（API 层）。通常我们通过 SLF4J 的 LoggerFactory.getLogger(...) 得到一个 Logger 并在代码中使用 logger.debug(...) / info(...) / error(...) 等方法产生日志事件

private static final Logger logger = LoggerFactory.getLogger(MyClass.class);通过工厂模式获取对象

职责

  • 根据配置（logger 的级别）决定是否要记录一条日志（这一步非常重要：用于避免不必要的性能开销）。

  • 在决定记录后，把日志信息封装成一个 ILoggingEvent（Logback 的事件对象），并把这个事件交给关联的 Appender 去处理。

Logger 的层级（Hierarchy）与 additivity

  • Logger 以包名/类名为层级（例如 com.example.service），子 logger 如果没有显式级别，会继承父 logger 的级别（“effective level”）。

  • Additivity：一个 logger 的事件除了发送给自己的 appender，也会向父 logger 传播（除非设置 additivity="false"），因此一个事件可能被多个 appender 处理（例如既写文件又输出到控制台）

二. Appender（日志的“出口” / 目标）

Appender 定义日志事件要被写到哪里：控制台、文件、远程 Socket、数据库、邮件、消息队列、第三方聚合服务等。Logback 自带很多常用 appender（ConsoleAppender、RollingFileAppender、SocketAppender、DBAppender 等），也可以自定义。

接收 ILoggingEvent，可能先通过 Filter 做筛选（允许/拒绝/中断），然后把事件格式化并写到目标（IO 流或网络）。

Appender 有生命周期（start() / stop()），在配置解析完成后会被启动。

xxxxxxxxxx
<appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
  <encoder>
    <pattern>[%d{HH:mm:ss}] %-5level %logger{36} - %msg%n</pattern>
  </encoder>
</appender>


<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
  <file>logs/app.log</file>
  <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
    <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
    <maxHistory>7</maxHistory>
  </rollingPolicy>
  <encoder>
    <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger - %msg%n</pattern>
  </encoder>
</appender>
<!-- 异步包装器（把 FILE 包起来） -->
<appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
  <appender-ref ref="FILE"/>
  <queueSize>512</queueSize>
</appender>

三.Layout / Encoder（日志的“格式化”）

概念差异（重要）

  • 在 Log4j 的时代“Layout”更常见；在 Logback 中 Appender 使用 Encoder 来输出字节（Encoder 接受事件并产生 byte[]），Encoder 内部通常可以包装一个 Layout（LayoutWrappingEncoder）。Logback 同时提供 PatternLayout / PatternLayoutEncoder 作为最常用的格式化方式。

  • 总结：Layout -> 生成字符串；Encoder -> 生成字节（可包装 Layout）。在配置文件中你通常会看到 <encoder><pattern>...</pattern></encoder>。

常用 pattern（PatternLayout）标记

  • %d{...}：时间（例如 %d{yyyy-MM-dd HH:mm:ss.SSS}）

  • %thread：线程名

  • %level：日志级别（INFO/DEBUG）

  • %logger{36}：logger 名，后缀长度限制

  • %msg：日志消息（参数化后的字符串）

  • %n：换行

  • %X{key} 或 %mdc：MDC（Mapped Diagnostic Context）中的值（用于请求级别上下文，如 requestId）

  • %ex / %throwable：异常堆栈信息

  • %caller / %C / %M / %L：调用者信息（类/方法/行号）——注意：获取调用堆栈开销较大

四、日志系统的“流水线”原理（从调用到落盘 — step by step）

  1. 应用代码调用：比如 logger.info("Order {} processed", orderId)。

  2. 快速级别检查：Logger 首先检查该级别是否被启用（有效级别）。如果不启用，直接返回（避免进一步开销）。

  3. 构建 ILoggingEvent：如果启用，Logger 把数据封装成 ILoggingEvent，包含时间戳、线程名、logger 名、级别、消息模板与参数、MDC、异常信息（如果有）等。

  4. TurboFilters（可选）：Logback 的 TurboFilter 可以在 Logger 级别之前拦截和决定是否接受、拒绝或中断日志事件（用于全局级别的快速判断）。

  5. 分发给 Appender：Logger 把事件交给自己绑定的 Appender 集合。若 additivity 为 true，事件也会传递到父 logger 的 appender。

  6. Appender 层过滤：每个 appender 可以有自己的 Filter，决定是否处理该事件。

  7. 格式化 & 写出：Appender 使用 Encoder（或 Layout）把事件格式化成字节流，写入目标（文件、控制台、网络）。在写出时会触发 IO 操作（这是日志性能瓶颈的常见点）。

  8. 异步 / 缓冲：若使用 AsyncAppender，写出过程放到单独线程中，主线程不会等待 IO（提升吞吐但增加日志丢失风险在极端 crash 场景）。

  9. 完成：日志落盘或发送成功（或失败并触发 appender 的错误处理策略）。

五、重要扩展概念（在项目中常用）

  • MDC（Mapped Diagnostic Context）：线程级别的 key-value，常用于把 requestId、userId 等上下文加入到所有同线程的日志中。使用：MDC.put("reqId","abc123")，Pattern 中用 %X{reqId}。别忘了在请求结束时 MDC.clear()。

  • Markers：用于给日志打标签，方便特定处理（例如只把带某 marker 的日志发到特定 appender）。

  • Filters 与 TurboFilter：可做更细粒度控制（比如只记录特定 IP、特定级别上下文的日志）。

  • Logback Context（LoggerContext）：整个 Logback 的运行上下文，配置/状态都在这里。可以通过编程方式动态改变配置。

  • JSON / 结构化日志：生产环境常用 JSON 输出（便于 ELK/EFK/数据仓库），可以使用第三方 encoder（如 logstash-logback-encoder）来输出 JSON。

接下来，我们详细讲每一块。

三、详细讲解每个部分

1️⃣ <property> —— 定义变量或格式

这个部分定义一些可复用的常量：

xxxxxxxxxx
<property name="LOG_PATTERN"
          value="[%d{yyyy-MM-dd HH:mm:ss.SSS}] [%thread] %-5level %logger{36} - %msg%n"/>

解释：

✅ 你可以理解为定义了一个“模板”，日志打印时会按照这个格式输出。

2️⃣ <appender> —— 日志输出的地方

Appender 决定日志输出到哪里，比如：

  • 控制台 (ConsoleAppender)

  • 文件 (FileAppender)

  • 异步输出 (AsyncAppender)

(1) 控制台输出

xxxxxxxxxx
<appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
        <pattern>${LOG_PATTERN}</pattern>
    </encoder>
</appender>

作用： 日志打印到 IDEA 的控制台。 关键点：

  • name="CONSOLE" 给它取个名字（后面引用用）

  • class="ConsoleAppender" 是控制台输出类

  • <encoder> 里定义输出格式

(2) 文件输出

xxxxxxxxxx
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/app.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
        <fileNamePattern>logs/app.%d{yyyy-MM-dd}.log</fileNamePattern>
        <maxHistory>7</maxHistory>
    </rollingPolicy>
    <encoder>
        <pattern>${LOG_PATTERN}</pattern>
    </encoder>
</appender>

解释：

  • 日志保存到 logs/app.log

  • 每天会滚动生成一个新文件（app.2025-10-11.log）

  • maxHistory=7 表示只保留最近 7 天的日志

(3) 异步日志

xxxxxxxxxx
<appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
    <appender-ref ref="FILE"/>
    <queueSize>1024</queueSize>
</appender>

解释：

  • 异步日志不会阻塞程序运行（性能更高）

  • ref="FILE" 表示它实际写入 FILE 日志

  • queueSize=1024 表示缓存队列大小

  • 一、AsyncAppender 是什么（一句话）

    AsyncAppender 是一个包装型的 appender：把日志事件放到一个队列里，由单独的后台线程异步取出并转发到它引用的真实 appender（如 FILE、CONSOLE 等）去写入 IO。 目的：把耗时的 IO 操作从业务线程剥离，降低对主线程延迟的影响，提高吞吐。

    二、这段配置在做什么（运行时流程）

    1. 应用线程调用 logger.info(...)。

    2. 如果该 logger 绑定了 ASYNC_FILE，AsyncAppender.doAppend(event) 会把 ILoggingEvent 放入内部队列（有时先对事件做“冻结/准备”，以保证延迟处理时数据不丢失）。

    3. 一个或多个专门的工作线程（由 AsyncAppender 管理）从队列中取事件并把它们按顺序传给 <appender-ref ref="FILE"/> 所指的真实 appender（例如 RollingFileAppender），由那个 appender 执行实际的格式化与 IO 写入。

    4. 主线程不会等待磁盘写入（除非队列已满并被配置为阻塞），因此业务线程延迟明显降低；代价是在 JVM 突然崩溃或进程非正常退出时，队列中尚未写出的日志可能丢失。

    三、主要配置项解释（常见且重要的 child 元素）

    • <appender-ref ref="FILE"/> 指定 AsyncAppender 在后台会把事件转发到名为 FILE 的 appender。可以引用多个 appender（比如同时转发到 FILE 和 CONSOLE）。

    • <queueSize>1024</queueSize> 内部队列的容量（即最多缓存多少条 ILoggingEvent）。

      • 影响：队列越大能缓冲更多突发日志，降低主线程被阻塞/丢日志的概率；但占用更多内存（事件对象、参数数组等）。

      • 如何选：视应用峰值日志量和内存情况设定。常见值有 256/512/1024；高吞吐场景可更大。不要随意设成非常大（几万），否则内存压力和 GC 会影响性能。

    • neverBlock（可选，boolean）

      xxxxxxxxxx
      <neverBlock>true</neverBlock>

      • 如果 neverBlock=true：当队列满时，主线程不会阻塞，而采用非阻塞方式尝试入队，入队失败时直接丢弃该日志（或按丢弃策略处理）。适合低延迟优先、允许少量日志丢失的场景（比如高频交易、低延迟服务）。

      • 如果 neverBlock=false（或不配置）：当队列满时，默认行为通常是使主线程阻塞直到队列有空位（这保证日志不丢，但增加延迟、可能导致系统被日志拖慢或出现倒挂）。

      • 选择建议：如果不能接受阻塞，设置 neverBlock=true 并配合丢弃策略；如果不能容忍丢日志，保持阻塞并合理增大 queueSize 和监控。

    • discardingThreshold（可选）

      • 作用：当队列占用达到某个阈值时，AsyncAppender 会优先丢弃低优先级的日志（如 TRACE/DEBUG），以保证重要级别（WARN/ERROR）能进入队列。

      • 你可以用它来控制“在队列拥塞时哪些日志可以被牺牲”。（不同版本实现细节略有差别，通常配合 neverBlock 使用）

    • includeCallerData（可选，boolean）

      xxxxxxxxxx
      <includeCallerData>true</includeCallerData>

      • 是否在入队前把“调用者信息”（类/方法/行号）捕获并保存在事件中。

      • 代价：采集调用者信息需要走栈（Throwable/StackTrace），非常耗时；在高吞吐场景建议关闭（false）。如果需要输出 %caller、%C、%M 等 pattern，必须打开它。

    • appender-ref 数量

      • 可以有多个 <appender-ref>，工作线程会把事件顺序地发给每个引用的 appender（也就是说，一个事件可以被同时写到多个目的地）。

3️⃣ <root> —— 根日志配置

xxxxxxxxxx
<root level="info">
    <appender-ref ref="CONSOLE"/>
    <appender-ref ref="ASYNC_FILE"/>
</root>

解释：

  • 设置所有日志的默认级别为 info

  • 所有日志都会输出到控制台和文件中

  • 级别从低到高：trace < debug < info < warn < error
